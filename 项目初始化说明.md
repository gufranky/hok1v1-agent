# 项目初始化说明

## 1. 项目是做什么的

这个仓库是一个 **《王者荣耀 1v1》强化学习训练样例工程**，核心目标是训练一个 1v1 英雄对战智能体。

仓库里有两套 agent 实现：

- `agent_ppo`：已经实现好的 PPO 版本，是当前可直接跑通、可直接分析的主实现。
- `agent_diy`：留给用户自己扩展的占位版本，目前大部分逻辑还是空壳。

从顶层配置看，默认算法是 `ppo`：

- `conf/app_conf_hok1v1.toml` 中，`train_one` 和 `train_two` 都指向 `algo = "ppo"`。
- `conf/algo_conf_hok1v1.toml` 中，`ppo` 的 `actor_agent / learner_agent / train_workflow` 都已经接到 `agent_ppo`。

所以，当前这个项目最值得关注的是 `agent_ppo` 目录。

## 2. 目录结构

### 顶层

- `train_test.py`

- `kaiwu.json`

- `agent_ppo/model/model.py`
  - PPO 模型定义与 loss 计算。
- `agent_ppo/algorithm/algorithm.py`

- `agent_ppo/feature/`
  - 特征构建、样本拼接、奖励计算。

  - PPO 相关超参数、输入输出维度、奖励权重。

## 3. 训练流程在做什么

PPO 的主流程在 `agent_ppo/workflow/train_workflow.py`，主线可以概括成：

1. 创建环境配置管理器 `EnvConfManager`。
2. 按轮转方式生成对局阵容。
   - 当前代码里英雄池是 `[112, 133]`，也就是两名候选英雄做两两组合轮换。
3. 每局开始前：
   - 更新对局配置；
   - 调用双方 agent 的 `init_config` 选择召唤师技能；
   - `env.reset()` 启动一局新环境；
   - `agent.reset()` 重置 LSTM 状态、奖励管理器和特征处理器。
4. 对局逐帧进行：
   - 环境给出 observation；
   - agent 做 `predict()` 或 `exploit()`；
   - 输出动作给 `env.step(actions)`；
   - 根据新帧状态计算 reward；
   - 训练局会把每一帧拼成样本，写入 `FrameCollector`。
5. 对局结束后：
   - 进行 GAE 计算；
   - 将样本打包成 learner 需要的格式；
   - 发给 learner 训练；
   - 定期保存模型。

### 对手类型

`agent_ppo/conf/train_env_conf.toml` 里当前配置：

- `opponent_agent = "selfplay"`

这表示默认是 **自博弈训练**。

另外也支持：

- `common_ai`：规则 AI
- 自定义模型 ID：从模型池中取对手

## 4. 输入是什么

这里的“输入”分三层看最清楚：

### 4.1 环境原始输入

环境每一帧会给 agent 一份 observation，至少包含这些关键字段：

- `frame_state`
  - 原始环境帧信息，里面有：
    - `hero_states`
    - `npc_states`
    - `frame_no`
- `legal_action`
  - 当前帧可执行动作掩码。
- `sub_action_mask`
  - 子动作掩码，后面会写进训练样本。
- 其他基础字段
  - 如 `camp`、`player_id` 等。

reward 不是环境直接给模型用的最终 reward，而是 agent 自己根据 `frame_state` 重新计算。

### 4.2 模型真正吃到的特征输入

模型真正吃的是 `FeatureProcess.process_feature()` 生成的 **10 维特征**。

在 `agent_ppo/conf/conf.py` 中：

- `DimConfig.DIM_OF_FEATURE = [10]`
- `Config.SERI_VEC_SPLIT_SHAPE = [(10,), (85,)]`

意思是：

- `10` 维是模型输入特征；
- `85` 维是合法动作掩码相关输入；
- 合起来每帧样本前半部分是 `10 + 85 = 95` 维。

### 4.3 10 维特征具体是什么

当前 PPO 版本特征非常小，只取了：

#### 英雄特征：3 维

来自 `hero_feature_config.ini`：

1. `is_hero_alive`
   - one-hot，当前配置只有 `1:eq`，实际展开后是 1 维
2. `location_x`
3. `location_z`

所以英雄部分总共是 `1 + 1 + 1 = 3` 维。

这里实际只处理 **己方主英雄** 的特征。

#### 敌方防御塔特征：7 维

来自 `organ_feature_config.ini`：

1. `is_alive`
2. `belong_to_main_camp`
3. `location_x`
4. `location_z`
5. `relative_location_x`
6. `relative_location_z`
7. `hp_rate`

所以防御塔部分总共 7 维。

这里实际只取 **敌方塔**，而且只取 `sub_type == 21` 的塔对象。

#### 合计

- 英雄：3 维
- 敌方塔：7 维
- 总计：10 维

### 4.4 训练样本里的完整输入

训练时不只是 10 维特征，还会把这些内容一起打包进样本：

- 特征 `feature`
- 合法动作 `legal_action`
- `reward_sum`
- `advantage`
- 动作 `action`
- 各动作头的历史概率 `prob`
- `sub_action`
- `is_train`
- LSTM hidden / cell

在 `agent_ppo/conf/conf.py` 中：

- `LSTM_TIME_STEPS = 16`
- `LSTM_UNIT_SIZE = 512`

所以 learner 用的是 **按 16 帧拼接后的序列样本**。

## 5. 奖励是什么

奖励逻辑在 `agent_ppo/feature/reward_process.py`。

当前实现非常简单，只用了两个 reward 项：

### 5.1 `tower_hp_point`

权重在 `agent_ppo/conf/conf.py`：

- `tower_hp_point: 5.0`

它表示双方塔血量归一化后的差分奖励，属于 **零和奖励**：

- 我方塔状态变好会加分；
- 敌方塔状态变好会减分；
- 实际计算的是“当前帧差值 - 上一帧差值”。

### 5.2 `forward`

权重：

- `forward: 0.01`

这个奖励用来鼓励前压。核心逻辑：

- 计算我方英雄到敌方塔的距离；
- 计算我方塔到敌方塔的距离；
- 如果英雄满血附近（`hp/max_hp > 0.99`）且比我方塔更靠前，就给一个前进奖励。

这个奖励不是零和项，而是直接取己方前进值。

### 5.3 最终 reward

每帧最终奖励：

```text
reward_sum = tower_hp_point * 5.0 + forward * 0.01
```

此外：

- `TIME_SCALE_ARG = 0`
  - 当前没有启用时间衰减。
- `FrameCollector` 会把 reward clip 到 `[-100, 100]`。
- 训练时再基于 `GAMMA=0.995`、`LAMDA=0.95` 计算 GAE advantage。

## 6. 输出是什么

这里“输出”也分三层：

### 6.1 模型直接输出

`agent_ppo/model/model.py` 中，模型输出：

- 6 个离散动作头 logits
- 1 个 value

动作头维度来自：

- `LABEL_SIZE_LIST = [12, 16, 16, 16, 16, 9]`

也就是说每帧输出：

1. 动作头 1：12 维
2. 动作头 2：16 维
3. 动作头 3：16 维
4. 动作头 4：16 维
5. 动作头 5：16 维
6. 动作头 6：9 维
7. 价值头：1 维

### 6.2 agent 给环境的动作输出

最终给环境的是一个 **6 维离散动作列表**。

默认空动作定义在 `agent_ppo/feature/definition.py`：

```python
NONE_ACTION = [0, 15, 15, 15, 15, 0]
```

说明环境期望的动作格式就是 6 个离散槽位。

### 6.3 训练和评估时的输出方式

- `predict()`
  - 训练时使用，按概率随机采样动作。
- `exploit()`
  - 评估时使用，选最大概率动作。

也就是说：

- 训练输出：随机采样动作
- 评估输出：贪心动作

### 6.4 最后一个动作头的特殊性

最后一个 9 维动作头不是完全独立的，它依赖第一个动作头。

代码在 `Agent._sample_masked_action()`：

- 前 5 个动作头先分别采样；
- 第 6 个头（target）会根据第 1 个动作头对应的 one-hot，再从目标合法动作里筛一遍；
- 所以第 6 个头是 **条件目标选择**。

这也是为什么 legal action 总维度不是简单的 `12+16+16+16+16+9=85`，而是：

- `12 + 16 + 16 + 16 + 16 + (9 * 12) = 168`

但在训练样本里，真正存进去的 `legal_action` 是经过 `_update_legal_action()` 裁剪后的版本，总长度仍然回到 `85`。

## 7. PPO learner 在优化什么

learner 入口在 `agent_ppo/algorithm/algorithm.py`。

主要优化三部分：

1. `value_loss`
   - 拟合 `reward_sum`
2. `policy_loss`
   - PPO clip loss
3. `entropy_loss`
   - 提高策略熵，鼓励探索

总损失：

```text
loss = value_cost + policy_cost + beta * entropy_cost
```

其中：

- `BETA_START = 0.025`
- `CLIP_PARAM = 0.2`

`agent_ppo/conf/train_env_conf.toml` 当前配置如下：

- `monitor_side = 0`
  - 监控蓝方

  - 自动切换监控视角
- `opponent_agent = "selfplay"`
  - 自博弈
- `eval_interval = 10`
  - 每 10 局评估一次

- 蓝红双方初始示例英雄：
  - `112`

但训练主流程里会通过 `lineup_iterator_roundrobin_camp_heroes([112, 133])` 轮换阵容，所以实际训练并不只固定在 112。

## 9. 怎么跑

### 快速自测入口

```powershell
python train_test.py
```

`train_test.py` 里当前默认：

```python
algorithm_name = "ppo"
```

如果你想切 DIY，需要先改成：

```python
algorithm_name = "diy"
```

但当前 `diy` 还没真正实现，能分析的重点仍然是 `ppo`。

### 当前能确认的事实

- 本仓库依赖外部 `kaiwudrl` 运行时。
- 仓库里没有看到完整依赖安装说明。
- 所以“完整训练能否直接在当前机器上运行”，还取决于外部环境是否已经配好。

## 10. 一句话总结

这个项目本质上是一个 **HOK 1v1 强化学习样例工程**：

- **项目做什么**：训练一个 1v1 对战智能体，当前主实现是 PPO。
- **输入是什么**：环境 observation + legal_action，其中模型核心特征是 10 维（己方英雄 3 维 + 敌方塔 7 维）。
- **奖励是什么**：塔血量差分奖励 + 满血前压奖励。
- **输出是什么**：6 个离散动作头组成的环境动作，以及 1 个 value 头。

如果你后面要继续深入，下一步最值得做的是：

1. 把 `agent_ppo` 跑一次 `train_test.py`
2. 再去看环境实际返回的 observation 样例
3. 最后再决定要不要扩输入特征和奖励项
